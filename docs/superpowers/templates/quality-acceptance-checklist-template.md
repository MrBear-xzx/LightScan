## 3. 开发门禁

### 3.1 架构与代码规范

- [ ] 模块边界清晰（控制面/执行面/插件层未发生跨层耦合）
- [ ] 新增核心模块具备类型标注与注释说明
- [ ] 关键流程具备结构化日志（至少含 `trace_id` / `task_id`）
- [ ] 配置项与常量统一管理，未出现硬编码敏感配置

### 3.2 插件质量

- [ ] 新增或变更插件实现完整契约：`discover/scan/normalize/health`
- [ ] 插件异常路径（超时/失败/重试）已覆盖
- [ ] 插件版本与兼容矩阵已登记
- [ ] 插件启停/灰度流程已验证

## 4. 测试门禁

### 4.1 测试执行结果

- [ ] 单元测试通过
- [ ] 服务测试通过
- [ ] API 测试通过
- [ ] 最小 e2e 冒烟测试通过
- [ ] 回归测试通过（含历史高频缺陷）

### 4.2 覆盖率与质量阈值

- [ ] 核心领域代码覆盖率 >= 80%
- [ ] 全仓总体覆盖率 >= 70%
- [ ] 本次缺陷修复均附带回归测试
- [ ] 无 P0/P1 未关闭缺陷

## 5. 数据与规则质量门禁

- [ ] 资产去重规则已验证（同资产多源合并正确）
- [ ] 漏洞去重规则已验证（重复告警聚合正确）
- [ ] 去重/评分规则已版本化并可回滚
- [ ] 风险评分权重变更有审计记录
- [ ] 标准化字段完整性校验通过（severity/confidence/fingerprint）
- [ ] 异常数据已进入隔离路径且可追踪

## 6. 发布门禁

### 6.1 交付物完整性

- [ ] 版本说明（Release Notes）完整
- [ ] 升级步骤完整且经演练
- [ ] 回滚步骤完整且经演练
- [ ] 兼容性说明完整（数据库、插件、接口）

### 6.2 变更风险控制

- [ ] CI 全绿（lint/test/build/security scan）
- [ ] 灰度发布方案明确
- [ ] 回滚开关已配置并验证
- [ ] 关键依赖版本锁定且可复现

## 7. 运行门禁

### 7.1 观测性

- [ ] 核心监控已接入：任务成功率、失败率、耗时、队列堆积、插件健康
- [ ] 核心告警已配置：任务失败激增、队列积压、插件不可用
- [ ] 日志可追踪完整调用链

### 7.2 稳定性与容量

- [ ] 基础压测/容量评估完成并有结论
- [ ] 关键场景下无明显性能退化
- [ ] 数据归档与索引治理策略已确认

## 8. 安全与合规门禁

- [ ] RBAC 与最小权限验证通过
- [ ] 审计日志覆盖关键操作（登录、策略变更、状态流转、插件启停）
- [ ] 日志脱敏检查通过（无密钥/令牌/密码泄露）
- [ ] 开源依赖许可证与漏洞检查通过
- [ ] 私有化交付合规项已核对（如等保/内审要求）

## 9. 业务验收指标核对（来自 PRD）

- [ ] 资产发现有效率 >= 90%（在约定样本集）
- [ ] 任务执行成功率 >= 95%

- [ ] 高危漏洞 MTTR 可统计并呈下降趋势
- [ ] 新增一个扫描插件接入周期 <= 5 个工作日

## 附录 B：严重级别定义（建议）

- `P0`：阻断发布或导致核心能力不可用
- `P1`：影响关键路径，需发布前修复
- `P2`：可发布后短期修复并跟踪
- `P3`：优化项，纳入后续迭代
