# Git 分支与协作流程规范

- 版本：v1.0
- 生效日期：2026-06-23
- 适用范围：LightScan 全部代码与文档协作

## 1. 分支模型

- `main`：稳定主分支，只接收通过验收的合并。
- `feature/*`：新功能开发分支。
- `fix/*`：缺陷修复分支。
- `hotfix/*`：线上紧急修复分支。
- `chore/*`：工程治理与非功能变更（文档、脚手架、CI）。

命名格式：
- `feature/<模块>-<简述>`
- `fix/<模块>-<问题>`
- `hotfix/<影响面>-<简述>`
- `chore/<类别>-<简述>`

示例：
- `feature/discovery-task-api`
- `fix/dedup-fingerprint-collision`
- `chore/docs-quality-checklist`

## 2. 开发流程

1. 从 `main` 拉取最新代码。
2. 创建个人工作分支。
3. 小步提交，保持每次提交可回滚、可理解。
4. 推送远程并发起 PR。
5. 通过代码评审与质量门禁后合并。
6. 合并后删除临时分支。

## 3. 合并策略

- 默认使用 `Squash and merge`，保证主分支历史简洁。
- 禁止直接向 `main` push。
- 至少 1 名 reviewer 通过（核心模块建议 2 名）。
- 存在未解决评论或失败检查时禁止合并。

## 4. 同步策略

- 开发周期内至少每日同步一次 `main`。
- 大型分支（>3 天）建议每日 rebase 或 merge main，减少冲突。
- 发生冲突必须在本分支解决，不在 `main` 直接修冲突。

## 5. 变更粒度要求

- 一个 PR 聚焦一个主题，避免“大杂烩”。
- 功能、重构、格式化尽量分离。
- 安全修复单独 PR，便于追溯与审计。

## 6. 发布与回滚

- 发布前必须关联 PRD/计划/验收清单。
- 每次发布必须有回滚路径。
- `hotfix` 修复后必须回合并到 `main`，并同步相关开发分支。

## 7. 禁止事项

- 禁止强推覆盖他人分支历史（`git push --force` 到共享分支）。
- 禁止绕过评审与门禁直接合并。
- 禁止提交敏感信息（密钥、令牌、密码、证书）。
