# AGENT.md（项目级协作约束）

- 生效日期：2026-06-23
- 适用范围：本仓库内所有 AI/人工协作任务

## 1. 语言与沟通

1. 默认使用中文回复。
2. 沟通要简洁明确，先结论后细节。
3. 关键决策必须给出可执行选项，不给空泛建议。

## 2. 当前阶段约束

1. 当前项目处于前期规划与治理完善阶段。
2. 未收到明确指令“开始写代码/进入实现”前，不进行业务代码开发。
3. 优先完成：PRD、实施计划、质量门禁、协作治理文档。

## 3. 执行顺序（必须遵守）

1. 先阅读并对齐以下文档：
- `docs/superpowers/specs/2026-06-23-vulnerability-analysis-platform-design.md`
- `docs/superpowers/plans/2026-06-23-vulnerability-analysis-platform-mvp.md`
- `docs/superpowers/templates/quality-acceptance-checklist-template.md`
- `docs/governance/git-workflow.md`
- `docs/governance/commit-convention.md`
2. 先做文档和方案，再做实现。
3. 任何实现任务都必须先有对应计划与验收标准。

## 4. 质量门禁（实现阶段强制）

1. 测试分层：单元、服务、API、最小 e2e 冒烟齐备。
2. 覆盖率目标：核心领域 >= 80%，整体 >= 70%。
3. CI 未通过不得合并。
4. 关键链路日志必须包含 `trace_id` / `task_id`。
5. 每个缺陷修复必须附回归测试。

## 5. Git 与提交规范

1. 禁止直接推送 `main`，必须走 PR。
2. 分支命名使用：`feature/*`、`fix/*`、`hotfix/*`、`chore/*`。
3. Commit 信息默认使用中文（含标题与必要说明）。
4. Commit 格式遵循仓库提交规范文档。
5. 默认 `Squash and merge`。

## 6. 安全与合规红线

1. 禁止提交密钥、令牌、密码、证书等敏感信息。
2. 日志中不得输出敏感字段明文。
3. 权限相关变更需明确评估影响与回滚路径。
4. 引入依赖需进行许可证与漏洞检查。

## 7. 变更边界

1. 不做与当前目标无关的重构。
2. 不擅自扩大需求范围。
3. 若发现冲突或高风险，先暂停并确认再继续。

## 8. 交付要求

1. 每次交付必须说明：改了什么、为什么改、如何验证。
2. 若未执行某项（如测试/部署），必须明确说明原因。
3. 文档类任务优先给出文件路径与关键变更摘要。
